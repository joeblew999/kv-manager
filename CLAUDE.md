# Notes for Claude — joeblew999/kv-manager

This is **joeblew999's fork** of [dreamcatcher-tv/kv-manager](https://github.com/dreamcatcher-tv/kv-manager) (or whichever upstream the fork tracks — see `git remote -v`).

Same overlay shape as [joeblew999/d1-manager](https://github.com/joeblew999/d1-manager). Read that fork's CLAUDE.md if you want the long version of the rationale.

## Why this fork exists

kv-manager is a self-hostable Cloudflare Worker that admins **every KV
namespace on the operator's account** (its API token is account-scoped).
It's a sibling tool to d1-manager — together they cover D1 and KV ops.

Forked because the upstream `wrangler.toml` couples to the author's domain
(`kv.adamic.tech`) and metadata D1 id. Templating + per-operator
`config/<env>.env` makes the deploy reproducible per fork. Upstream code
under `worker/`, `src/`, `package.json`, etc. is **untouched** — bumps land
via `git merge upstream/main`.

## Operator runbook (one-time)

```sh
# 1. Configure
cp config/template.env config/production.env
$EDITOR config/production.env       # set CF_SUBDOMAIN + OPERATOR_EMAIL

# 2. CF API token (one-time per CF account, in dashboard).
#    Required scopes — same as d1-manager fork:
#      Account: Workers Scripts/D1/R2/KV: Edit, Account Settings: Read,
#               Access: Apps and Policies: Edit, Access: Organizations: Read
#    If you've already done this for d1-manager (and stashed the token
#    in fnox), you can skip — that token is reused here.

# 3. Stash canonical secrets in fnox (one-time per machine).
#    `mise run secrets:status` to check what's already there.
fnox set --global -p keychain CLOUDFLARE_API_TOKEN
fnox set --global -p keychain CLOUDFLARE_ACCOUNT_ID

# 4. Provision Cloudflare resources (R2 bucket + metadata D1 + schema).
#    Idempotent. Writes the metadata D1 id back into config/production.env.
mise run cf:provision

# 5. Set up Cloudflare Access (App + GitHub IdP + allow policy). Fully automated:
#    - Reuses CF_ACCESS_TEAM_DOMAIN + CF_ACCESS_GITHUB_IDP_ID from fnox
#      (cached when d1-manager was deployed — no re-discovery needed).
#    - Creates the Access App for kv-manager.<subdomain>.workers.dev.
#    - Adds an allow-by-email policy for OPERATOR_EMAIL.
#    - Writes KV_MANAGER_POLICY_AUD to fnox.
mise run cf:access:setup

# 6. Push the four secrets to the deployed Worker.
#    Maps fnox canonical names -> kv-manager Worker env names:
#      CLOUDFLARE_ACCOUNT_ID  -> ACCOUNT_ID
#      CLOUDFLARE_API_TOKEN   -> API_KEY
#      CF_ACCESS_TEAM_DOMAIN  -> TEAM_DOMAIN
#      KV_MANAGER_POLICY_AUD  -> POLICY_AUD
mise run secrets:put-cf

# 7. Build + deploy.
mise run 10-deploy

# 8. Verify everything end-to-end.
mise run prove:all
```

After deploy, visit `https://kv-manager.<CF_SUBDOMAIN>.workers.dev` — CF
Access challenges, GitHub OAuth, you land in kv-manager's UI listing every
KV namespace on your account.

## Decisions you should not reopen

### Why a fork instead of patching at deploy time

Same reasoning as d1-manager — see [its CLAUDE.md](https://github.com/joeblew999/d1-manager/blob/main/CLAUDE.md) for the long version.

### Why KV_MANAGER_POLICY_AUD specifically (not POLICY_AUD)

Each Access-gated app has its OWN audience tag. fnox keys must be per-app
to avoid d1-manager + kv-manager overwriting each other's POLICY_AUD. The
mise.toml's `secrets:put-cf` task maps `KV_MANAGER_POLICY_AUD` (fnox) →
`POLICY_AUD` (Worker secret name expected by kv-manager's worker code).

### Why the same CLOUDFLARE_API_TOKEN works across both forks

Account-scoped CF API tokens manage every D1, R2, KV, Worker, and Access
App on the account. d1-manager and kv-manager don't conflict because they
admin different resource types. The token only had to be edited once
(adding Access scope) when d1-manager was first deployed.

## Upgrading from upstream

```sh
git remote add upstream https://github.com/dreamcatcher-tv/kv-manager.git    # adjust if different
git fetch upstream
git merge upstream/main
mise run wrangler:gen          # re-render wrangler.toml from template
mise run 10b-redeploy          # if no rebuild needed
# OR
mise run 10-deploy             # full rebuild + deploy
mise run prove:all
```

If the upstream `wrangler.toml` adds new bindings or removes existing
ones, mirror those changes in `wrangler.toml.template` (use `$VAR` for
anything that varies per operator).
